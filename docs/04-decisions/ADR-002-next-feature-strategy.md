# ADR-002: Next Feature Strategy — Insight Loop as Retention Driver

Generated by /office-hours on 2026-06-01
Branch: main
Status: APPROVED

---

## Context

Kora v1 is a local-first personal tracking app with 10 active features (Todo,
Routines, Water, Mood, Gratitude, Calendar, Challenges, Breathing, Affirmations,
Timer) and a Voice/NLP layer. As of 2026-06-01:

- All three original P0 blockers are resolved.
- A signed release APK (204.5 MB) builds and verifies cleanly.
- `flutter analyze` passes with zero issues.
- 46 entity-level unit tests pass, but there are no provider, repository, or
  widget integration tests — test coverage is not release-meaningful.
- An `AnalysisService` singleton (Pearson correlation, mood × water, mood by
  weekday, mood by time of day — 216 lines at
  `lib/features/analytics/data/services/analysis_service.dart`) is fully
  implemented and reads Hive boxes directly. The companion `AnalysisCard`
  widget (`lib/features/analytics/presentation/widgets/analysis_card.dart`)
  already calls the service in its own `_loadAnalysis()` method and cycles
  through insights with a Timer. Neither is referenced from the Home screen.
- The Home screen (`lib/features/home/presentation/pages/home_page.dart:176`)
  shows an `InsightsCard` (defined in
  `lib/features/todo/presentation/widgets/dashboard_widgets.dart:869`).
  `InsightsCard` is a data-driven contextual tip that reads live from
  `WaterProvider`, `MoodProvider`, `RoutinesProvider`, and `TodoProvider` and
  picks the most relevant nudge (hydration reminder, mood check-in, routine
  progress, etc.). It is NOT static text — it is contextual advice. What is
  missing is a second layer: pattern-based insight derived from multi-day
  correlation analysis. This is what `AnalysisCard` provides.
- Voice command parsing works end-to-end, but unimplemented command types
  return a silent no-op. Users receive false success feedback (P1-1).
- The affirmation feature has two registered storage paths: a Hive repository
  (`lib/features/affirmations/data/repositories/`) with all four domain
  usecases (`GetRecordings`, `SaveRecording`, `DeleteRecording`,
  `SaveAffirmationSession`) that are never called anywhere in the app, and an
  active SharedPreferences/files path used by `AffirmationProvider`. The Hive
  adapter self-registers via the repository file, but `main.dart` never imports
  that repository, so typeId 12 is effectively never registered at startup.
  This entire Hive path is dead code.

**The core problem:** `InsightsCard` answers "what should I do right now?"
`AnalysisCard` answers "what does my data say about me over time?" Both are
valuable. Right now only the first is visible. Users can track mood, water,
routines, and gratitude for two weeks without the app ever surfacing a
multi-day pattern. The "aha moment" — when the app says something true and
specific about YOUR data across multiple days — is the event that converts a
curious installer into a daily user. That moment is built and waiting; it just
needs to be placed on the screen.

---

## Options Considered

### Option A: Add AnalysisCard to Home Screen (Minimal Viable)

The `AnalysisCard` widget already calls `AnalysisService()` directly, handles
the empty state by returning `SizedBox.shrink()`, and cycles through multiple
insights with a Timer. No new Provider, no major refactor. The only code
change is placing `const AnalysisCard()` in `home_page.dart`, preceded by the
voice feedback fix and 2 focused safety tests.

- **Effort:** S (~2–3 days total including voice fix and tests; the widget
  placement itself is ~1 hour)
- **User value:** HIGH — first time the app surfaces multi-day pattern insight
- **Retention impact:** HIGH — personalized correlation data is the event that
  forms a habit loop in tracking apps
- **Technical risk:** LOW — `AnalysisCard` is already written and handles all
  null/empty cases; `SizedBox.shrink()` means it is invisible until data is
  sufficient
- **Release readiness:** Resolves P1-2 (analytics not wired to any screen)

### Option B: Release-Safe Foundation First

Write a full provider test suite (MoodProvider, RoutinesProvider,
CalendarProvider) plus one smoke widget test before touching any user-facing
code. Analytics wiring follows after.

- **Effort:** L (~3 weeks)
- **User value:** ZERO during development period
- **Retention impact:** ZERO (prevents future churn from bugs, but does not
  acquire or retain users now)
- **Technical risk:** Lowest possible for public launch
- **Release readiness:** Maximum — every provider bug caught before first user

### Option C: Weekly Insight Push Notification

Wire analytics computation to a scheduled weekly notification (Sunday, 8 PM).
The notification fires only when |r| ≥ 0.3 and data ≥ 3 aligned days.

Key constraint: `flutter_local_notifications` requires static notification
content at scheduling time — it cannot generate dynamic content at fire time.
The notification body must be computed at the time `zonedSchedule` is called.
This means the weekly insight notification must be rescheduled on each app
open (computing the current correlation and scheduling for the next Sunday).
Users who do not open the app before the scheduled Sunday will not receive
a fresh notification.

- **Effort:** S-M (~3–4 days, primarily due to the reschedule-on-open logic)
- **User value:** HIGH for users who do not open the app daily
- **Retention impact:** HIGH — re-engagement trigger for drifting users
- **Technical risk:** MEDIUM — reschedule-on-open logic needs care to avoid
  double-scheduling; existing NotificationService already handles repeating
  weekly notifications for routines
- **Release readiness:** No change to Home screen content; notification
  scheduling complexity is contained

---

## Decision

**Proceed with Option A + C combined: add `AnalysisCard` to the Home screen
(A) and wire a reschedule-on-open weekly insight push notification (C) in a
single sprint.**

The combined sprint is ~5–6 days and delivers:
1. Voice command trust restoration (P1-1 fix) — 0.5 days
2. Two targeted safety tests for the analytics path — 0.5 days
3. `AnalysisCard` added to Home screen alongside existing `InsightsCard` — 1 hour
4. Weekly insight push notification (rescheduled on app open) — 2–3 days
5. Remove dead affirmation Hive path (parallel, ~2–3 hours) — 0.5 days

---

## Reasoning

### Why the two Home screen cards coexist

`InsightsCard` and `AnalysisCard` serve distinct purposes:

- `InsightsCard` is a **daily action nudge** — "You've hit 50% of your water
  goal, try to drink 400ml more." It reads live session data and tells you
  what to do in the next hour.
- `AnalysisCard` is a **multi-day pattern insight** — "On days you drink more
  water, your mood tends to be higher (r=0.72 over 14 days)." It reads
  historical correlation data and tells you something true about yourself over
  time.

Both should be visible. `AnalysisCard` should be placed below `InsightsCard`
in the scroll order. Because `AnalysisCard` returns `SizedBox.shrink()` when
data is insufficient (fewer than 3 aligned days), it is invisible to new users
and only appears once they have meaningful history. No empty-state design work
is needed — the widget already handles this correctly.

### Why AnalysisCard needs no new Provider

`AnalysisCard` already calls `AnalysisService()` directly in `_loadAnalysis()`
within its own `initState`. It manages its own async loading state and exposes
nothing to the outside tree. Adding it to Home is a one-line widget placement.
Creating a wrapping `AnalyticsProvider` would add complexity without benefit
for v1 — the card refreshes on each rebuild of its parent widget, which is
sufficient for a personal tracking app where data changes infrequently.

The only wiring needed: `const AnalysisCard()` in `home_page.dart`.

### Why voice fix precedes analytics placement

P1-1 (silent voice command failure) is not a polish issue. The voice layer
is positioned as a power feature. When it silently does nothing on an
unimplemented command, users learn to distrust it. That distrust bleeds into
their perception of the whole app. Fixing it takes half a day and removes a
negative signal before the positive signal (analytics) ships.

### Why 2 targeted safety tests before analytics placement

The `AnalysisService` reads Hive boxes via `await Hive.openBox()`. Because
Hive safely returns the already-open box when called again, there is no
production race condition — `AnalysisCard` initializes in the widget tree
after `main.dart` completes, and the boxes are already open by then.

However, the analytics path has never been exercised by any test, and the
service has no null-safety guarantees documented. Two tests are sufficient
to make the placement safe:

1. `analyzeWaterMoodCorrelation()` returns `null` when the raw count of
   `MoodEntry` records OR `WaterIntake` records in their respective Hive boxes
   is fewer than 3 (the guard at `analysis_service.dart:43` checks
   `moods.length < 3 || waterIntakes.length < 3` — raw box count, not aligned
   days). Seed sparse data where one or both boxes have 2 entries.
2. `analyzeWaterMoodCorrelation()` returns a non-null string with a
   recognizable correlation interpretation when given 5+ `MoodEntry` +
   `WaterIntake` records where the dates align and the water amounts track
   the mood levels (i.e., both datasets are constructed to produce a
   meaningful r-value).

Note on test 2: `_calculatePearsonCorrelation` is private. The test must
exercise it through the public `analyzeWaterMoodCorrelation()` method using
Hive in-memory setup (`hive_test` package or `Hive.init()` with a temp
directory) and seeded `MoodEntry` + `WaterIntake` records with a known
correlation. The Hive box names used in `AnalysisService`
(`'mood_entries'`, `'water_intake'`) must match those opened in `main.dart`
— the test implicitly verifies this consistency.

### Why notification rescheduling-on-open is the right approach

`flutter_local_notifications` requires static content at scheduling time via
`zonedSchedule`. The notification body cannot be generated dynamically at fire
time. The practical path for a weekly insight notification is:

On each app open (in `main.dart` after full initialization or in a lifecycle
hook), compute the current correlation via `AnalysisService`, and if the
result is non-null and the correlation is meaningful (|r| ≥ 0.3), schedule a
new weekly notification for the next Sunday at 8 PM, overwriting any
previously scheduled one. If the result is null (insufficient data or
correlation below threshold), cancel any pending weekly insight notification.

This means:
- Users who open the app at any point during the week receive an insight
  notification that Sunday with data current as of their last open.
- Users who do not open the app before Sunday receive no notification (or the
  notification from the previous week if data was sufficient then).
- There is no background fetch or WorkManager dependency.

The user name for the notification body is read from `FlutterSecureStorage`
key `'user_name'` (same key used by `home_page.dart:58`). This read must be
awaited before calling `zonedSchedule`.

### Why affirmation Hive path deletion is in this sprint

The dead affirmation Hive path includes:
- Domain usecases: `GetRecordings`, `SaveRecording`, `DeleteRecording`,
  `SaveAffirmationSession` — never called
- Data repository: `AffirmationLocalRepository` — never called
- Domain repository interface — never referenced outside the repository
- Hive adapter self-registration in `affirmation_repository.dart:12` —
  never triggered because `main.dart` never imports this repository

The canonical path is SharedPreferences/files (used by the active
`AffirmationProvider`). Removing the dead path eliminates schema ambiguity,
frees typeId 12, and reduces confusion for future work. The `AffirmationSession`
model and its generated `*.g.dart` file can also be deleted unless they are
referenced elsewhere (they are not — grep confirms zero usage outside the
dead repository). Total cleanup: delete ~5 files, update hive-typeids.md.

## Prioritized Roadmap

### Phase 1 — Trust Restoration (0.5 day)

**Fix P1-1: Voice command silent no-op**

File: `lib/core/services/command_parser.dart`

When a parsed intent has no handler registered, surface an explicit
"Command not supported yet" message via the VoiceCommandService back to
`VoiceCommandSheet` rather than silently returning success. This is a UX fix:
no new functionality, just honest feedback for the `not_implemented` branches.

Acceptance: saying "start meditation" (unimplemented) shows a visible
"Command not supported yet" response in the voice sheet instead of the current
silent success state.

### Phase 2 — Analytics Safety Net (0.5 day)

**Write 2 targeted tests for the analytics path**

File: new `test/features/analytics/analysis_service_test.dart`

1. `analyzeWaterMoodCorrelation()` returns `null` when mood or water entries
   total fewer than 3 aligned days. Use Hive in-memory setup with seeded
   sparse data.
2. `analyzeWaterMoodCorrelation()` returns a non-null string when given 5+
   aligned `MoodEntry` + `WaterIntake` records with a known positive
   correlation. Verify the returned string contains a human-readable
   interpretation (smoke test on output format).

Test setup: use the `hive_test` package or `Hive.init()` with a temp
directory, register `MoodEntryAdapter` and `WaterIntakeAdapter`, seed test
data programmatically, and call `await AnalysisService().analyzeWaterMoodCorrelation()`.

### Phase 3 — Home Screen Insight Layer (1–2 hours)

**Add `AnalysisCard` to Home screen**

File: `lib/features/home/presentation/pages/home_page.dart`

Place `const AnalysisCard()` below the existing `InsightsCard()` in the
Home screen scroll content. No other code changes required:

- `InsightsCard` remains and continues to serve its role (contextual daily
  action nudge based on live provider data).
- `AnalysisCard` adds the multi-day pattern layer (correlation-based insight).
- `AnalysisCard` returns `SizedBox.shrink()` when data is insufficient, so
  it is invisible to users until they have enough history. No empty-state
  design work needed.
- No Provider wrapper needed — `AnalysisCard` manages its own async state.

Before placing the card, manually verify the UX copy produced by
`AnalysisService.analyzeWaterMoodCorrelation()`. The service was written as
a backend utility; the string format has never been reviewed as user-facing
text. If it reads poorly (e.g., "💡 Strong Positive Correlation Detected:
Pearson r=0.72"), revise the string template in `analysis_service.dart`
before shipping.

Acceptance: after 3+ days of aligned mood + water records, `AnalysisCard` is
visible on the Home screen below `InsightsCard` and shows a correlation
insight that reflects the actual recorded data.

### Phase 4 — Weekly Insight Push (2–3 days)

**Reschedule-on-open weekly insight notification**

Files:
- new `lib/features/analytics/domain/usecases/compute_weekly_insight.dart`
- `lib/core/services/notification_service.dart` (add
  `scheduleWeeklyInsight(String? insight, String userName)`)
- `lib/main.dart` (call reschedule after full initialization)

Steps:

1. Create `ComputeWeeklyInsight` use case (returns `String?`):
   - Calls `AnalysisService().analyzeWaterMoodCorrelation()` only.
   - Returns `null` if result is `null` (insufficient data or weak correlation).
   - Note: this use case covers water/mood correlation only. The Home screen
     `AnalysisCard` shows three insight types (water/mood correlation,
     mood-by-weekday, mood-by-time-of-day). The notification intentionally
     covers only the water/mood correlation — it is the most actionable pattern
     (directly tied to a behavior the user can change today) and is the insight
     with the most data, since both mood and water are tracked daily. If a user
     has a strong weekday or time-of-day pattern but no water/mood correlation,
     they will not receive a notification for those patterns in v1. This is
     accepted scope for the first notification; v2 can cycle through all three
     insight types.

2. Add `scheduleWeeklyInsight(String insight, String userName)` to
   `NotificationService`:
   - Cancels any previously scheduled weekly insight notification (use a
     dedicated notification ID, e.g., 9001).
   - Calls `flutterLocalNotificationsPlugin.zonedSchedule(...)` with the
     next Sunday at 20:00 in the device timezone.
   - Uses `DateTimeComponents.dayOfWeekAndTime` for the repeating match.
   - Body: `"$userName, this week: $insight"`

3. In `lib/main.dart`, after full initialization (after
   `NotificationService.init()` and all box opens complete), call:
   ```
   final insight = await ComputeWeeklyInsight()();
   final name = await FlutterSecureStorage().read(key: 'user_name') ?? 'there';
   if (insight != null) {
     await NotificationService().scheduleWeeklyInsight(insight, name);
   } else {
     await NotificationService().cancelNotification(9001);
   }
   ```

Acceptance: after 3+ days of aligned mood + water data, opening the app
schedules (or reschedules) a Sunday 8 PM notification with a correlation
insight. Tapping the notification navigates to the Home screen. No
notification is scheduled when data is insufficient.

### Phase 5 — Remove Dead Affirmation Hive Path (0.5 day)

**Delete the unused Hive repository and domain layer for affirmations**

Files to delete:
- `lib/features/affirmations/data/repositories/affirmation_local_repository.dart`
- `lib/features/affirmations/data/repositories/affirmation_repository.dart`
- `lib/features/affirmations/domain/repositories/affirmation_repository.dart`
- `lib/features/affirmations/domain/usecases/get_recordings.dart`
- `lib/features/affirmations/domain/usecases/save_recording.dart`
- `lib/features/affirmations/domain/usecases/delete_recording.dart`
- `lib/features/affirmations/domain/usecases/save_affirmation_session.dart`
- `lib/features/affirmations/data/models/affirmation_session.dart`
- `lib/features/affirmations/data/models/affirmation_session.g.dart`

Files to update:
- `docs/02-architecture/hive-typeids.md`: mark typeId 12 as removed.
- `pubspec.yaml` / `build_runner`: no change needed (generated files deleted
  directly).

Verify with `grep -r "AffirmationSession\|affirmation_repository" lib/`
that no references remain outside the deleted files before committing.
The active `AffirmationProvider` uses SharedPreferences and local files — it
has no dependency on any of the deleted files.

### Phase 6 — Provider Test Coverage (Post-Sprint, ~1 week)

**Write provider-level tests for the 3 highest-risk paths**

Priority order:
1. `MoodProvider` — data-loss risk (the box-open failure fix in P0-1 has no
   test; a regression would silently delete user mood history)
2. `RoutinesProvider` — notification scheduling logic (incorrect reminder
   times mis-schedule user alarms silently)
3. `CalendarProvider` — reminder time calculation (offset math errors are
   invisible until a user misses an event)

These tests do not require a full Hive environment — mock the data sources at
the Provider boundary using interfaces that already exist in the domain layer.

Phase 6 is not deferred indefinitely. It begins in the week after the insight
loop sprint ships and completes before a public Play Store listing is created.

## Success Criteria

1. **Home screen shows `AnalysisCard` below `InsightsCard`** for any user with
   3+ aligned mood + water records. Card is invisible (not shown) for new users.
2. **Weekly Sunday notification fires** with a correlation insight after 3+
   days of data. No notification fires on sparse data. Opening the app
   reschedules the notification with fresh data.
3. **Voice command silent failures are eliminated.** All unimplemented command
   types surface a visible "Command not supported yet" response.
4. **2 analytics safety tests pass** in the CI suite before the Home screen
   change merges.
5. **Dead affirmation Hive path is deleted.** TypeId 12 freed and documented.
   Zero references to `AffirmationSession` remain in active code.

## Consequences

### Positive

- Kora acquires its first multi-day feedback loop. The app now says something
  specific about the user's data pattern, not just today's nudge.
- Home screen gains a second insight layer without displacing the existing
  contextual tip card — both serve users.
- Weekly push re-engages drifting users without requiring them to already be
  active.
- P1-1 and P1-2 are resolved.
- Affirmation storage ambiguity is eliminated, freeing typeId 12.

### Accepted Technical Debt

- Provider test coverage remains thin for Mood, Routines, Calendar, Breathing,
  Affirmations, and Challenges after this sprint. This is **accepted
  consciously** and scheduled for Phase 6 before public Play Store listing.
- `BreathingProvider` (666 lines) and `AffirmationProvider` (648 lines)
  remain unsplit. Splitting them is valid debt reduction but not a retention
  or release blocker. Defer to post-v1.
- `main.dart` (437 lines) remains monolithic. Deferred — extracting adapter
  registration is correct but not urgent.
- `AnalysisCard` retains its direct `AnalysisService()` call rather than going
  through a Provider. This is intentional for v1 simplicity; if analytics
  data needs to be shared across multiple widgets in a future release, a
  `ChangeNotifierProvider` wrapper would be the correct refactor.

### Risks

- The weekly notification reschedules on app open, so users who open the app
  between the last scheduled Sunday and any given Monday through Saturday will
  see a notification with data current as of their most recent open. Users
  who only open on Sundays after 8 PM may miss the notification for that week.
  This is acceptable for v1.
- If `AnalysisService` string output format is not reviewed before shipping
  (Phase 3 note), the Home screen may show technically correct but poorly
  phrased insight copy. The assignment below addresses this.
- Phase 6 provider tests are committed follow-on, not optional backlog. If
  they slip past the public launch date, any MoodProvider or WaterProvider
  regression could corrupt the analytics data users see on their Home screen.

### What Is Not Changing

- Local-only architecture: no cloud, no accounts, no export in this sprint.
- Navigation pattern: all pushes via `AppRoute` / `AppFadeRoute`.
- Hive type ID registry: no new type IDs needed for analytics.
- `InsightsCard` contextual tip behavior is unchanged.
- iOS: not in scope for v1.

---

## The Assignment

Before placing `const AnalysisCard()` on the Home screen, open the app on a
real device (or emulator with seeded data), add 5+ days of mood + water
entries, then trigger `AnalysisService().analyzeWaterMoodCorrelation()` from
a temporary debug tap or a unit test and read the output string out loud.

If it sounds like a data report, rewrite it to sound like a friend noticing
something about you. The copy in `analysis_service.dart` was written for an
internal service; this is the first user-facing text in Kora that says
something specific about the user's own data. It deserves to be good.
